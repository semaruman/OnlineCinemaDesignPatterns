# Моделирование отправки уведомлений и подписок онлайн-кинотеатра
> Цель: отработать на практике паттерны decorator и observer, применить маршрутизацию (MapGet, MapPost), понять устройство minimal api приложений
## Суть приложения
  Есть сериалы. На каждый сериал подписаны пользователи. Когда администратор от какого-то сериала отправляет уведомления, эти уведомления приходят подписчикам данного сериала (применение паттерна observer). При создании уведомления, администратор может выбирать несколько типов сразу. Из-за этого получается одно большое составное уведомление (применение паттерна декоратор). База данных отсутствует

## ✨ Возможности API

| Метод | Запрос | Функционал |
|-----------|-------------|-------------|
| **GET** | /serials | получение списка сериалов
| **GET** | /serials/id | получение информации о конкретном сериале
| **GET** | /serials/id/subscribers | получение подписчиков сириала
| **GET** | /users | получение всех пользователей
| **GET** | /users/id | получение информации о конкретном пользователе
| **GET** | /users/id/serials | получение сериалов пользователя
| **GET** | /users/id/notifications | получение уведомлений пользователя
| **POST** | /users/id/subscribe/serialId | пидписаться на сериал
| **POST** | /users/id/unsubscribe/serialId | отписаться от сериала
| **POST** | /users/add | добавление пользователя
| **POST** | /serials/id/addnotification | добавление уведомления для конкретного сериала

## 🧠 Что использовал

| Компонент | Пояснение |
|-----------|-------------|
| **Observer** |	Реализуете механизм "издатель-подписчик" без жесткой связки классов |
| **Decorator**	| Составное уведомление собирается как матрёшка из предыдущих |
| **Minimal API**	| Легковесные endpoints без контроллеров |
| **In-Memory хранение** |	Проектирование системы без БД (идеально для моего прототипа) |
| **HTML CSS JS** |	Для отражения работоспособности api |
